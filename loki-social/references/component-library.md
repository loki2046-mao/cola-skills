# Component Library

This library includes 108 reusable components. Combine them by content type and page role, but avoid nested-card clutter. A 7-9 page carousel should normally use 18-32 components total.

## Cover Components

1. **C01 Pain Hook Cover**: large pain sentence + tiny context + one evidence object.
2. **C02 Tension Cover**: one concrete tension without using "不是 X，是 Y"; pair the claim with one proof note.
3. **C03 Recruiter View Cover**: hiring/manager viewpoint + resume fragment.
4. **C04 Product Proof Cover**: product name secondary, result primary.
5. **C05 Number Promise Cover**: "7 天 / 3 个方向 / 5 个错误" when numbers are real.
6. **C06 Quote Cover**: one strong quote from article + source label.
7. **C07 Screenshot Cover**: screenshot crop dominates 40-60% of page.
8. **C08 Question Cover**: direct user question + answer hint.
9. **C09 Search-Bar Cover**: a search-like input containing the reader's exact anxiety.
10. **C10 Folder Cover**: desktop folder/file stack that implies a real deliverable.
11. **C11 Notebook Cover**: handwritten margin + typed headline for warm student topics.
12. **C12 Desk Scene Cover**: realistic work surface with documents, highlights, and result note.
13. **C13 Alert Cover**: warning strip + one concrete mistake, not generic fear.
14. **C14 Result Cover**: final artifact preview first, explanation second.
15. **C15 Split-View Cover**: two competing interpretations of the same material.
16. **C16 Minimal Object Cover**: one resume/page/app window as the only visual anchor.

## Component Selection Rules

- Use at least one evidence component in pages 3-5 for tool/Skill posts.
- Use at least one comparison component for transformation posts.
- Use annotation components for critique-heavy posts; they create human judgment.
- Use workflow components only after the reader understands why they should care.
- Use CTA component at the end, not as the opening unless the user asks for a tutorial.
- Do not use more than two dense data/table components in one carousel.
- Do not repeat the same cover or evidence layout in adjacent pages.

## Core Recipe Details

Use these recipes as structure hints when building HTML. They are intentionally compact; adapt spacing and typography to the selected style mode.

### R01 Bright Hook Cover
- Components: C01/C03/C10 + A03 + E14.
- Structure: top metadata, huge human sentence, one black or color signal tag, one visible object stack.
- Use when: source has a clear pain and a concrete artifact.
- Avoid: centered title only, old paper background, vague tool name as the main hook.

### R02 Recruiter Audit Page
- Components: E02 + B06 + A02 + E20.
- Structure: title, report/screenshot window, 3-row audit table, one caption explaining why the table matters.
- Use when: resume, job search, JD, HR, manager, or reviewer judgment is central.
- Avoid: tiny columns, decorative table with unreadable cells, unsupported claims.

### R03 Before/After Proof
- Components: B01 + E11 + A01.
- Structure: left original material, right improved version, one highlighted changed result.
- Use when: article has transformation, rewrite, workflow improvement, or output quality change.
- Avoid: comparing two equally long paragraphs; compress before putting in image.

### R04 Prompt / Input Artifact
- Components: E03 + E15 + W13.
- Structure: user input block, requested output block, 4-6 tiny chips showing required fields.
- Use when: the reader needs to know how to call a Skill or workflow.
- Avoid: long prompt walls; move full prompt to caption or pinned comment.

### R05 Evidence Window
- Components: E01/E04/E10 + A13 + E20.
- Structure: app/report window, readable excerpt, 1-3 callouts, one line explaining the evidence.
- Use when: user asks to include screenshots or show a tool result.
- Avoid: cropping half a screenshot; if source is unavailable, label as "demo excerpt" or "mock evidence".

### R06 Workflow Rail
- Components: W01/W02/W16 + I03.
- Structure: 3-5 vertical or staggered steps, each with one verb and one output.
- Use when: converting install/tutorial/action plan into a saveable page.
- Avoid: shell-command overload; long commands go to caption/comment.

### R07 Case Proof
- Components: K01/K02/K04/K05.
- Structure: persona/context, raw material, direction or output, one result note.
- Use when: long article has examples or sample users.
- Avoid: generic persona without concrete material.

### R08 Closing CTA
- Components: K06 + I01/P04.
- Structure: one final thesis, one next action, where to get the tool or what to comment.
- Use when: final page needs to convert saves/comments.
- Avoid: sales banner energy and repeated feature list.

## Failure Patterns To Reject

- One template repeated for every page with only text changed.
- Pale yellow, beige, or old-paper background dominating the set.
- Large bright rectangles that look like AI-generated placeholder design.
- Empty space that has no counterweight from type, screenshot, or visual object.
- Quote marks or slogan tropes as the main visual.
- Mock screenshots presented as real screenshots.
- Evidence cropped halfway through text.
- More than two consecutive pages with only title + paragraph + card.

# 微信公众号文章排版

这个 skill 专门处理“我给你文章内容和图片，你负责把它排成公众号后台可用成品”的任务。

默认目标不是只给建议，而是直接产出可执行结果。

## 内容保真铁律（最高优先级）

**排版时不修改用户的文字内容和图片内容。** 这条高于所有其他规则。

- **不改写文字**：原文的每一个字都必须原样保留，包括措辞、语气、口语化表达、错别字（除非用户明确要求改错别字）。排版只做视觉结构化（分段、加标题层级、加强调样式），不做文字编辑。
- **不增删内容**：不添加原文没有的段落、句子、观点。不删除原文的段落、句子、括号旁白。原文有的内容必须全部保留，原文没有的内容绝不无中生有。
- **不改图片**：不裁剪、不调色、不压缩、不替换用户的图片。图片按原顺序、原归属插入对应位置。图片缺失时用占位组件标注，不用别的图顶替。
- **唯一允许的调整**：
  1. 段落拆分与合并（把一个大段拆成几个视觉段，或把几个碎段合成一个视觉段——但文字内容不变）
  2. 标题层级推断（把原文的加粗文本识别为小标题等——但不改文字）
  3. 强调样式添加（给原文已有的词语加下划线/加粗/高亮——但不改文字）
  4. 标点全角化（半角标点转全角，这是排版规范不是内容修改）
  5. 章节编号添加（01/02/03 等序号是排版元素，不是内容）
  6. 图注添加（给图片加一句说明性图注——但这是新增排版元素，不改原文文字）
- **如果原文有明显错误**（事实错误、错别字），不要自行修正，保持原文原样，可以在交付时提醒用户注意。

## 默认行为

- 默认输出适合微信公众号后台粘贴或手动套入的成品排版。
- 默认做单栏排版，不做复杂栅格、多列、悬浮、绝对定位。
- 默认优先保证阅读体验、结构清晰、审美统一，再考虑装饰感。
- 默认把“深浅色兼容”当成硬约束，而不是额外加分项。
- 当用户明确要求“滑动画廊”时，允许切换到 `SVG 交互排版模式`
- 当用户明确接受“整篇都写成 SVG”时，允许切换到 `整篇 SVG-first 模式`

## 适用输入

- 完整正文
- 半成品草稿
- 用户给出的章节结构
- 若干配图、截图、海报、封面图
- 只有文字没有图片的文章
- 只有图片顺序没有插图说明的素材包

## 推荐交付格式

这个 skill 默认不要求用户先手写 `Markdown`。

更符合实际的输入方式应该是：

1. `飞书文档 / WPS 笔记`
   - 默认推荐
   - 用户只需要在原文档里大致标出标题、正文、配图和特殊块
   - 这是最贴近真实工作流的交付方式
2. `结构化 Markdown + 图片文件夹`
   - 适合批量化、标准化生产
   - 不是默认前提，只是备选增强方案
3. `纯文本 + 素材包`
   - 可以做
   - 但必须接受我对标题层级、图片归位、模块拆分做较多推断

### 文档原生交付方式

如果你给我的是飞书文档或 WPS 笔记，最理想状态不是“重新整理成 Markdown”，而是在原文档里满足下面四件事：

1. `标题层级大致清楚`
   - 一级标题、小标题、正文不要混在一起
   - 强调句不要伪装成标题
2. `图片归属大致清楚`
   - 图片尽量放在对应段落或小节下面
   - 连续多张图默认视为同一组，除非上下文明显分开
3. `特殊块可识别`
   - 高亮块、引用块、代码块、提示词块，默认可视为特殊排版区域
   - 超长高亮块，默认优先理解为“长文本缩略卡片”
4. `顺序可信`
   - 文档里的先后顺序，默认就是成稿的叙事顺序

### 文档里建议保留的轻标记

不需要你额外写一份 `md`，只需要在飞书或 WPS 里尽量保留这些表达习惯：

- 标题就用真正的标题样式，不要全靠加粗冒充
- 一段正文下面紧跟的图片，默认属于这一段或这一小节
- 如果几张图必须同组，尽量连续放在一起
- 如果某段超长文字需要压缩展示，优先放进高亮块、引用块或单独卡片里
- 如果有“第一点 / 第二点 / 第三点”或“第一层 / 第二层 / 第三层”，尽量一行一条写清楚
- 如果某张图必须单独出现，最好在它前后保留一个独立段落，不要夹在组图中间

### 这个 skill 对原生文档的默认理解

当输入是飞书文档或 WPS 笔记时，默认按下面逻辑理解内容：

1. 真正的标题样式
   - 识别为章节标题
2. 普通正文段落
   - 识别为正文模块
3. 高亮块 / 引用块 / 明显独立的大段文本
   - 优先识别为重点块或长文本缩略卡片
4. 连续出现的多张图片
   - 优先识别为组图
   - 如果横竖比例混杂，按横版组和竖版组拆开
5. “第一点、第二点、第三点 / 第一步、第二步、第三步 / 第一层、第二层、第三层”
   - 优先识别为序列卡片或分栏模块

### 什么时候才需要额外提供 `article.md`

只有下面这些情况，才建议补一份结构化文本：

1. 文档本身非常乱，标题和正文样式没有区分
2. 图片很多，但归属关系完全不清楚
3. 同一篇里有大量“候选图”，但并不是都要用
4. 你希望把这套流程批量化、重复化

### `article.md` 作为增强方案

如果你愿意给结构化文本，那么准备两个东西：

- `article.md`
- `images/`

其中 `article.md` 至少要包含四类信息：

1. `标题层级`
   - 哪些是主标题
   - 哪些是小标题
   - 哪些只是正文里的强调句，不要误当标题
2. `段落结构`
   - 每个小节的正文内容
   - 哪些内容是长文本、提示词、引用、步骤、清单
3. `图片归属`
   - 哪张图跟哪个小节走
   - 多张图是否属于同一组画廊
4. `特殊排版意图`
   - 哪些地方要做长文本缩略卡片
   - 哪些地方要做“第一层/第二层/第三层”序列卡片
   - 哪些图片必须单图展示，哪些允许做组图

### `article.md` 建议标注语法

在 `article.md` 里，优先使用这种简单标记：

```md
# 文章总标题

> 风格：白底、局部橙黄色、偏杂志感
> 模式：原生稳定 / SVG-first / SVG 交互画廊

## 小节标题

这里是正文第一段。

这里是正文第二段。

[配图]
- img-01.jpg

## 第二个小节

这里是正文。

[组图]
- img-02.jpg
- img-03.jpg
- img-04.jpg

[序列卡片]
1. 第一层：一句说明
2. 第二层：一句说明
3. 第三层：一句说明

[长文本卡片]
这里放长提示词、长摘录、长方法论。
[/长文本卡片]
```

### 图片文件夹规则

- 所有图片都放在同一个 `images/` 文件夹里
- 文件名要稳定、清楚，最好带顺序号
- 推荐命名：
  - `img-01-cover.jpg`
  - `img-02-chat.png`
  - `img-03-case-a.jpg`
- 如果有“候选图”但不一定用，单独放进 `images/optional/`
- 如果一张图必须放在某一段后面，不要只给文件名，必须在正文里标出来

### 对图片归位的最低要求

如果你不想写完整 Markdown，至少也要给我这三类信息：

1. 哪些文字是标题，哪些不是
2. 哪几张图属于哪一个小节
3. 哪些图是单图，哪些图可以并成同一组

只要这三件事明确，我就能比较稳定地调用这个 skill 做新排版。

### 面向 WPS / 飞书的实际调用原则

如果用户说“去处理我 WPS 笔记里的某一篇”或“去排我飞书文档里的这篇”，默认不要要求用户先转格式，而是直接：

1. 读取文档原内容
2. 识别标题、正文、图片顺序和特殊块
3. 推断哪些地方适合单图、组图、长文本卡片、序列卡片
4. 仅在图片归属或章节层级明显混乱时，再向用户补问最少量信息

## 最终产出目标

至少做到下面四件事：

1. 视觉上干净、耐看、有统一审美
2. 结构上清楚，不堆砌复杂装饰
3. 配色有主题，但不过度依赖大面积底色
4. 在微信公众号浅色模式和深色模式下都尽量保持稳定可读

## 工作流程

1. 先判断文章类型：`教程 / 观点 / 清单 / 故事 / 测评 / 资讯总结 / 情绪表达`
2. 再判断文章节奏：是“平铺叙述”、还是“章节推进”、还是“重点卡片型”
3. 再判断实现模式：
   - `原生稳定模式`：适合纯后台 HTML 排版
   - `SVG 交互模式`：适合需要滑动组图、交互画廊的文章
   - `整篇 SVG-first 模式`：适合用户接受整篇文章以 SVG 视觉结构输出
4. **选主题**：读 [主题索引](./references/theme-index.md)（主题信息单一来源），据文章题材推荐最契合的主题，让用户确认。用户已指定主题则直接用。默认推荐「清醒青灰」。详见 theme-index 的选择建议。
5. **读组件库（两份）**：(1) 读所选主题的组件库 `references/theme-{标识}.md`；(2) 同时读通用增量库 `references/common-components.md`（代码块/图片GIF/小标签标题，所有主题共用）。后续生成完全依据这两份组件库，HTML 一律从中取、不要手写。`references/wechat-safe-layout-recipes.md` 作为补充参考（配色、深浅色兼容规则、SVG 排版原则、图片编排规则、包边规则、视频说明卡、文末关注引导图等）。
6. 规划图片插入点：
   - 封面图或最强视觉图优先放在开头引子后
   - 信息截图放在对应解释段之后
   - 连续多张图只在确实属于同一组内容时才连排
   - 图片过多时，优先保留能推进理解的图，不要把文章变成相册
7. **按配方选组件组合 → 装配 HTML**：查所选主题库的「文章类型→组件组合配方」表，按文章类型确定核心组件组合与点缀组件。然后依主题库的「完整文章模板骨架」装配，把每个内容元素替换为对应组件。一篇文章只用所选主题的那一套组件，不跨主题混用。
8. **智能排版处理**（必须做）：
   - **章节自动编号**：按章节出现顺序分配 01/02/03…，末章用 `∞`
   - **正文关键词逐段下划线**：对每个正文段落主动找出 1-3 个最重要的短语，用该主题的下划线 CSS 标记。即使原文没有任何加粗也要主动加下划线
   - **中文全角标点**：正文标点一律用全角（，。！？：；“”‘’（）—— ……），生成 HTML 时就直接写弯引号，不事后替换。代码块/行内代码内保持原样
   - **签名区去重合并**：原文末尾若已有作者签名段，识别并并入唯一的签名区/CTA 卡片，不重复生成
9. 生成最终结果：
   - 原生稳定模式：输出完整 HTML（纯 `<section>…</section>` 正文片段，不要包 `<!DOCTYPE>`/`<html>`/`<head>`/`<body>`）
   - SVG 交互模式：输出 `SVG/嵌入结构 + 图片资源规划 + 使用说明`
   - 整篇 SVG-first 模式：输出 `整篇 SVG 结构 + 分段说明 + 资源要求`
   - 如果图片需要在后台手动上传，输出“文字排版 HTML + 图片插入计划”
   - 不知道图片真实地址时，绝不编造 `src`
10. 最后自检一次层级、间距、颜色和深浅色兼容性
11. 运行校验脚本，ERROR 清零才交付（见下方「校验步骤」）
12. 如果使用 SVG 交互模式，必须提醒做手机端预览验证
13. 如果使用整篇 SVG-first 模式，必须提醒做后台粘贴验证和手机端预览验证

## 深浅色兼容规则

这部分必须严格执行。

### 1. 正文优先靠字体层级，不靠大面积底色

- 大段正文默认尽量不要写死 `color`
- 通过字号、字重、段间距、标题、分隔来建立层级
- 如果必须指定正文色，使用柔和深灰，不要用纯黑

### 2. 色彩主要用于“点缀位”

- 颜色优先放在：标题标签、分隔线、编号、短提示条、细边框
- 不要让重要长文本依赖彩色底块才能看清
- 大面积高饱和背景在深色模式下风险很高，尽量不用

### 3. 避免这些高风险做法

- 避免纯黑 `#000000` 和纯白 `#FFFFFF`
- 避免荧光色、高饱和黄底、大面积深底浅字
- 避免长段文字放在实心彩色卡片里
- 避免复杂阴影、拟物边框、花哨纹理

### 4. 兼容优先级

- 第一优先级：正文可读
- 第二优先级：章节层级清晰
- 第三优先级：视觉精致

如果三者冲突，宁可少装饰，也不要牺牲阅读。

## SVG-first 使用原则

- 当用户明确接受“整篇写成 SVG”时，可以把文章的主要视觉层全部交给 SVG 结构处理
- 这种模式更适合：
  - 视觉感要求高的专题文章
  - 图片驱动型文章
  - 需要强统一风格的品牌感内容
  - 多图画廊、分段卡片、节奏型长文
- 这种模式要特别注意：
  - 文字不能排得过小
  - 段落不能过密
  - 不能只顾设计感而牺牲阅读效率
  - 必须保留清晰的章节分段
- 如果文章是超长深度正文，默认仍优先考虑“正文原生 + 局部 SVG”的混合方案，除非用户明确要求整篇都 SVG 化

## 结构规则

- 全文模块控制在 `4-6` 种核心模块以内
- 一篇文章里最多使用 `1` 套主色 + `1` 套辅助浅底
- 一级标题样式保持统一，不能每节都换样式
- 列表、引用、总结块可以变化，但要共享同一色系
- 相邻两个模块之间要有明显留白，不要挤成一坨

## 图片编排规则

- 图片不是装饰填空，必须服务于理解、情绪或节奏
- 截图类图片前后应有一句解释或小标题承接
- 情绪图/氛围图不要连续出现超过两张
- 所有图片默认做成圆角矩形，且同一篇文章内圆角与包边风格必须统一
- 每张图都要判断它承担的是哪种角色：
  - `封面图`
  - `解释图`
  - `案例图`
  - `对比图`
  - `收尾氛围图`
- 没有必要时不强行给每张图都写图注
- 图注只在“信息密度高”或“需要提醒读者看什么”时启用
- 默认优先读取“原文档里实际出现的图片节点”，不要把下载目录里的所有素材都当成正文配图
- 如果原文档可读取到的图片数量与素材目录不一致，以原文档实际插入顺序为准
- 只有在原文图片节点读取不完整时，才允许把本地素材目录当作候选池，并明确说明这是回退推断

## 图片展示模式

### 1. 单图

- 单图必须单独展现
- 默认使用“独立主图”方式，不和别的内容挤在同一视觉盒子里
- 图片外观默认是：圆角矩形 + 统一包边

### 2. 双图及以上

- 双图及以上默认进入 `画廊模式`
- 目标观感是“同一组图片被统一包装”，而不是简单连续堆图
- 如果用户明确要求滑动效果，优先进入 `SVG 交互画廊`
- 如果当前任务只允许输出微信公众号后台可直接粘贴的纯 HTML，则自动降级为“静态画廊感排版”

### 3. 画廊实现优先级

1. `SVG 交互画廊`
   - 用户明确要求“左右滑动”时优先启用
   - 允许使用 `SVG/公众号兼容交互结构/导入式代码方案`
   - 适合案例组图、步骤截图、前后对比图
2. `静态画廊感排版`
   - 用于纯公众号后台 HTML 成品
   - 通过统一外框、组标题、图序、缩略节奏、留白关系做出“画廊感”
   - 这是默认保底方案，优先保证发出去稳定

如果使用 SVG 交互画廊，必须在结果里标注“需手机预览验证”。

## 包边规则

- 同一篇文章内，所有图片必须使用同一套包边语言
- 包边可根据文章风格调整，但整篇内必须统一以下参数：
  - 边框颜色
  - 边框粗细
  - 圆角大小
  - 是否带外层浅底衬托
- 包边只能起到“精修感”和“收边感”，不能喧宾夺主
- 默认优先使用细边框，不用厚重相框感边框
- 如果正文已经有很多强调模块，图片包边要进一步收敛

## 输出格式

默认按这个顺序输出：

1. `排版定位`
   - 用 2-4 句话说明这篇文章的风格、配色和结构选择
2. `图片插入计划`
   - 如果图片不能直接带 URL，就列出“哪一张图插在哪一段后面”
3. `可直接使用的 HTML`
   - 放在一个 `html` 代码块里
   - 输出最终成品，不要混入解释性注释

如果用户明确说“只给我成品”，就只输出最终 HTML。

如果用了“静态画廊感排版”，在 `图片插入计划` 里注明哪些图片属于同一组。
如果用了 `SVG 交互模式`，在输出里额外给出：

1. `交互模式说明`
2. `资源要求`
3. `手机预览验证提醒`

如果用了 `整篇 SVG-first 模式`，在输出里额外给出：

1. `SVG 排版定位`
2. `资源清单`
3. `粘贴/预览验证提醒`

## 视频处理规则

- 如果原文里有视频，默认不要伪造“可直接包住微信原生视频卡片的自定义边框”
- 微信后台原生插入视频后，平台通常会生成自己的视频卡片结构，因此不能假设外层自定义框还能完整保留
- 稳定做法是：
  1. 先输出一个 `视频说明卡 / 视频封面卡`
  2. 再在其下方预留“后台插入视频”的位置
  3. 真正的视频由用户在微信后台手动插入
- 也就是说：保留的是“视频前的设计框”和“视频说明模块”，不是强行包住微信原生视频播放器本体
- 如果用户明确要求文中出现视频位，最终成品里必须有一个稳定的视频模块占位

## 文末关注引导图规则

- 每一篇文章的最后，默认必须追加一张 `关注引导图`
- 这张图不属于可选装饰，而是硬性收尾模块
- 如果原文档最后已经有引导关注图，优先使用原文档里的那一张
- 如果原文档图片读取不完整，必须向用户确认这张固定引导图的来源，或者使用用户预先指定的通用引导图
- 不允许因为正文重排而把这张图漏掉
- 这张图必须出现在全文最后一个正文模块之后
- 当前默认固定尾图路径：
  - `assets/signature.png`（相对于本 SKILL.md 所在目录）
- 如果用户没有另行指定尾图，默认使用这张图作为每篇文章的文末关注引导图

## HTML 生成规则

- 使用简单、稳妥、后台兼容性高的标签：`section`、`p`、`h2`、`h3`、`strong`、`span`、`img`
- 样式以内联 `style` 为主
- 不依赖外部 CSS、`class`、脚本、伪元素
- 不使用复杂表格来做视觉排版
- 不输出多余占位符文案，比如“这里插图”
- 不要把全文包成一个大色块容器
- 不要伪造图片链接、图片尺寸或图片说明
- 不要在纯 HTML 方案里伪造“可滑动交互”
- 不要把“第三方编辑器导出的效果”误写成“原生后台裸 HTML 效果”
- 不要把整篇 SVG 做成只有设计感、没有阅读节奏的海报长图

### `<span leaf="">` 包裹铁律（必须执行）

这是公众号粘贴后样式不丢失的命门。公众号编辑器会重写未包裹的文字节点，导致样式整片丢失。

- **所有文字节点**必须用 `<span leaf="">文字</span>` 包裹
- 包括标题、正文、标签、图注、引用、提示——所有可见文字
- 装饰性空元素（圆点、渐变分割线、装饰短横、时间线竖线）内部必须放 `<span leaf=""><br></span>` 占位，否则微信会剥掉样式
- **不要把 `font-size`/`border-bottom` 打在 `<strong>` 上**，也不要在同一个 `<p>` 里混多个不同 `font-size`——微信编辑器会自动“纠正”导致样式被重写。正确做法：拆成多个 `<p>`，每个 `<p>` 只有一个字号；高亮样式统一挂在外层 `<span>` 上
- 例外：`<img>` 标签本身不需要包裹，但其父容器需要存在

### 不用虚线框做强调

- 强调用**左竖条**或**药丸标签**，不要用四周虚线框（`border: dashed`）包标题或正文
- 虚线框笨重、抢戏，不符合“清爽不脏”的排版目标
- **唯一例外**：`待补素材占位`（此处需要插入图片/视频/录屏但尚未提供）可以用居中虚线框，因为它的语义就是“占位/待补”，与正文强调不冲突
- 主题库中明确定义为风格特征的虚线组件（如引用框）按主题库用法正常使用，不算违规

### 图片尺寸规则

- `<img>` 一律用 `max-width:100%;height:auto;display:block;margin:0 auto`
- 按图片自身尺寸显示并居中，大图缩到容器宽，小图保持原尺寸
- **不用 `width:100%`**——会把小图也拉伸变糊
- 只有表格、封面卡、流程图这类布局元素才允许 `width:100%`

### 图片 base64 内嵌规则（复制粘贴兼容）

公众号后台没有开放图片上传 API，但用户可以通过浏览器全选复制把图片一起带过去——前提是图片数据在剪贴板里。相对路径的图片在 Safari/Chrome 里虽然能显示，但复制时图片数据不一定跟随剪贴板。

**解决方案**：生成最终 HTML 时，把所有图片内嵌为 base64 data URI，使 HTML 完全自包含。这样浏览器全选复制时，文字 + 图片 + 样式全部一起进剪贴板，粘贴到公众号后台后图片自动上传。

**执行时机**：在 validate 校验通过后、交付前，作为最后一步。

**执行方式**：
```bash
# 把 HTML 中所有 src 路径的图片替换为 base64 data URI
python3 ./scripts/embed_images.py <HTML路径>
```
输出 `-embedded.html` 文件，交付这个文件给用户。

**注意**：
- 文件不存在或为 0 字节的图片跳过（不内嵌），保留原 src
- http/https 开头的远程图片不内嵌
- 已经是 data: 开头的跳过
- 内嵌后文件会变大（约原图片总大小 × 1.37），公众号粘贴不受影响

### 代码块/长文本规则

- **不用 `white-space:pre`**——它会把 HTML 源码里 span 前的缩进和行间换行原样渲染成大左缩进 + 空行
- 多行代码块用“每行一个 `<p style="margin:0">`”的写法
- 缩进用全角空格 `　`，行距靠 `line-height` 控制
- 长文本卡片内部如果需要保留格式，用 `white-space:pre-wrap` + `word-break:break-word`，不用 `pre`

## 组件选择建议

优先从组件库 `references/components.md` 的「文章类型→组件组合参考」表选组件组合。核心组件构成本篇的排版主旋律，点缀组件按内容出现处使用。

所有文章共用固定结构：开头引子 + 章节标题 + 总结收尾 + 签名区 + 文末关注引导图。

代码块、图片、小标签标题等全部在 `components.md` 里。

**松弛感原则**：同一种内容换着排，不要连续5段都用同一种正文段落——中间插短句、居中句、引用、斜体旁白，让视觉有起伏。

## 校验步骤（强制）

生成最终 HTML 后，**必须运行校验脚本**，ERROR 清零才算完成：

```bash
# 用脚本所在 skill 的绝对路径调用，HTML 参数用其实际路径
./scripts/validate_gzh_html.py <生成的.html 的实际路径>
```

它确定性地检查公众号平台禁用项和 `<span leaf>` 包裹率。
- 报 ERROR 就回到生成步骤修，**ERROR 必须清零**
- 半角标点 WARNING 同样要修复到 0 再交付（这是实际使用中最高频的返工点）

如果修改了组件库参考文件（`references/` 下的文件），也应跑源头检查：

```bash
python3 ./scripts/component_lint.py ./
```

## 质量检查清单

输出前逐项确认：

- 是否一眼能看懂文章层级
- 是否颜色统一，不东一块西一块
- 是否有任何长文本压在彩色底上
- 是否有过于花哨、像模板站成品的装饰
- 是否图片密度过高
- 是否即使去掉所有背景色，结构依然成立
- 是否单图和组图的展示逻辑明确
- 是否整篇图片的圆角与包边完全统一
- 是否所有文字节点都已用 `<span leaf="">` 包裹
- 是否所有图片都用了 `max-width:100%` 而非 `width:100%`
- 是否有任何强调用了四周虚线框（应该改用左竖条/药丸标签）
- 是否有代码块用了 `white-space:pre`（应该改用每行一个 `<p>` + 全角空格缩进）
- 是否已运行校验脚本且 ERROR 清零
- 如果是 SVG 交互模式，是否已经提醒手机端验证
- 如果是整篇 SVG-first 模式，是否已经提醒后台粘贴与手机端双重验证

## 关键原则

这类任务的核心不是“做出很多样式”，而是“做出公众号后台里真正稳、真正好看的样式”。

最终成品要像人工精修过的排版，不像 AI 随手拼的模板。

## 添加新配色的规范

新配色添加到 `references/theme-index.md` 的 12 套表中（同时同步到 `loki-social/references/themes.md`），登记一行（主题名 / 底色 / 主色 / 辅色 / 文字色 / 适合场景 / 深浅），并跑 `python3 scripts/component_lint.py .` 确认 0 ERROR。

组件库不需要改——配色与组件分离，新配色只影响变量替换。
